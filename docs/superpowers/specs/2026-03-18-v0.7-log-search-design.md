# agent-procs v0.7 Design Spec

Log search and filtering for efficient agent-driven debugging.

## Goals

1. Let agents find specific log output (errors, patterns, debug prints) without pulling entire logs into context
2. Enable time-based log slicing so agents can see "what happened since I ran that test"
3. Make filtering available in both CLI and TUI with consistent semantics

## Theme

Evolve agent-procs from "process supervisor" to "process observatory." The daemon already captures and indexes all output â€” v0.7 adds the query layer so agents (and humans in the TUI) can ask targeted questions instead of scrolling through raw logs.

## Feature 1: `--grep` Flag

### CLI Surface

```bash
# Substring match (default)
agent-procs logs server --grep "error"

# Regex match (opt-in, consistent with `wait --regex`)
agent-procs logs server --grep "error|panic|warn" --regex

# Combine with existing flags
agent-procs logs server --grep "error" --tail 50
agent-procs logs --all --grep "timeout" --stderr
```

### Semantics

`--grep` applies a filter on log lines. Only matching lines are returned.

**Where filtering runs:**
- **Non-follow mode:** Client-side in `src/cli/logs.rs`. The daemon is not involved for non-follow log reads (`src/daemon/server.rs` rejects non-follow log requests). Currently the CLI reads logs via `tail_file()` (plain `BufReader` over `File::open`) with no index support. **v0.7 refactors this path** to use `DiskLogReader` (moved from `src/tui/` to `src/daemon/` or `src/` as a shared module) for LIDX-backed random access, which is required for efficient `--grep --tail N` (scan backward for last N matches) and `--since` (seek to estimated line position).
- **Follow mode:** Daemon-side in the broadcast subscriber loop. The daemon filters lines before sending them over the socket. New protocol fields (`grep`, `regex`) are only needed for `--follow` requests.

This split matches the existing architecture and avoids routing non-follow requests through the daemon.

**Substring mode** (default): case-sensitive `contains()` check. Same as the TUI filter and `wait --until`.

**Regex mode** (`--regex`): full regex syntax via the `regex` crate. Same as `wait --until --regex`. If the regex fails to compile, the command returns an error (exit code 1) with a diagnostic message â€” no silent fallback.

**Stream semantics (CLI only):** `--grep` searches whichever stream is selected. By default, stdout only. With `--stderr`, stderr only. These semantics apply uniformly whether searching a single process or with `--all` â€” e.g., `--all --stderr --grep "error"` searches each process's stderr independently, then groups results by process. There is no single-process "both streams" mode on the CLI today â€” that remains a TUI-only feature (`e` key to cycle).

**TUI stream semantics are unchanged.** The TUI's live log subscription (`Request::Logs { all: true, stderr: false }`) has existing special-case handling in the daemon that delivers both streams regardless of the `stderr` flag. The `--grep`/`--regex` follow-mode filter in the daemon must preserve this behavior â€” it filters by pattern only, not by stream. Stream filtering in the TUI continues to be controlled by the `e` key (StreamMode cycle) in the rendering layer, not the subscription layer.

### Filter Pipeline

The filter pipeline runs in this order:
1. Time window (`--since`) narrows the line range
2. Pattern match (`--grep`) selects matching lines
3. Windowing (`--tail N`) takes the last N results
4. Context expansion (`--context N`) adds surrounding lines around each match

This means `--tail 5 --grep error` returns the 5 most recent errors, then context is expanded around those 5 matches. The total output may exceed 5 lines when context is included, but only 5 actual matches are selected.

The CLI may need to scan backward from the end of the log to find the last N matches efficiently. `DiskLogReader` already supports random access via LIDX â€” scan from the end, collect matches, stop when N are found.

### Composition with `--follow`

`--follow --grep` streams only matching lines live. The existing broadcast subscriber loop in the logs handler gets a filter predicate. Non-matching lines are skipped before writing to the client.

`--since` is ignored in `--follow` mode (it only applies to historical lines). `--tail` with `--follow` replays the last N matching historical lines before switching to live mode (consistent with current `--follow --tail` behavior, but filtered).

### Composition with `--all`

`--all --grep` searches across all processes. Each process's logs get the same filter pipeline applied independently, then results are grouped by process (not interleaved chronologically â€” there is no cross-process sequence counter, so true chronological ordering is not possible). This matches the current `--all` behavior which concatenates per-process output.

The JSON output includes the `process` field so the agent knows which process each match came from.

### Protocol Changes

Extend `Request::Logs` with new fields for follow-mode filtering only (all `#[serde(default)]` for backward compat):

```rust
Request::Logs {
    target: Option<String>,
    tail: usize,
    follow: bool,
    stderr: bool,
    all: bool,
    timeout_secs: Option<u64>,
    lines: Option<usize>,
    // New in v0.7 (used only in follow mode):
    #[serde(default)]
    grep: Option<String>,
    #[serde(default)]
    regex: bool,
}
```

Note: `since` and `context` are not sent over the protocol â€” they are resolved client-side for non-follow mode. For follow mode, `--since` is ignored and `--context` is not applicable.

No new request variants. No new response variants â€” filtered lines are returned through the same streaming path that exists today.

### Files Changed

- `src/main.rs`: Add `--grep`, `--regex`, `--since`, `--context`, `--json` to `Logs` command args.
- `src/cli/logs.rs`: Rewrite non-follow path to use `DiskLogReader` instead of `tail_file()`. Implement filter pipeline (since â†’ grep â†’ tail â†’ context). Pass `grep`/`regex` to daemon for follow mode. Format JSONL output when `--json` is set.
- `src/protocol.rs`: Add `grep`, `regex` fields to `Request::Logs`.
- `src/daemon/server.rs`: Apply grep filter to follow-mode broadcast subscriber.
- Move `src/tui/disk_log_reader.rs` to a shared location (e.g., `src/daemon/disk_log_reader.rs` or `src/disk_log_reader.rs`) so both CLI and TUI can use it. Update `src/tui/app.rs` imports accordingly.

## Feature 2: `--since` Flag

### CLI Surface

```bash
# Wall-clock durations
agent-procs logs server --since 30s
agent-procs logs server --since 5m
agent-procs logs server --since 2h

# Event-relative
agent-procs logs server --since restart    # since last restart
agent-procs logs server --since start      # since process started (all output)

# Combines with grep
agent-procs logs --all --grep "error" --since 1m
```

### Time Specs

| Spec | Meaning |
|------|---------|
| `Ns` | N seconds ago |
| `Nm` | N minutes ago |
| `Nh` | N hours ago |
| `restart` | Since last restart annotation in logs |
| `start` | All retained history (current + rotated segments) |

### Prerequisite: Restart Markers on All Paths

**v0.7 adds a restart annotation to all restart paths, written before any child output.**

Currently only auto-restart and watch-restart write supervisor annotations, and they write them *after* spawn via the supervisor channel â€” creating a race where fast child output can precede the marker. Manual restart writes no annotation at all.

**Fix:** Add an optional `initial_annotation: Option<String>` parameter to `spawn_process()`. When set, `capture_output()` writes this line to the log file and LIDX index *before* entering the `select!` loop that reads from the child's pipe. This guarantees the marker is the first line in the log, preceding any child output. No race.

The callers pass the annotation at spawn time:
- `respawn_in_place()` via the actor's `handle_auto_restart()`: passes `Some("[agent-procs] Restarted (exit N, attempt M/K)")`
- `respawn_in_place()` via the actor's `handle_watch_restart()`: passes `Some("[agent-procs] File changed, restarted")`
- `restart_process()` via the actor's `PmCommand::Restart` handler: passes `Some("[agent-procs] Restarted (manual)")`
- First spawn (no restart): passes `None`

The existing post-spawn supervisor channel sends for auto/watch restart annotations are removed â€” the initial_annotation replaces them.

With this change, ALL restart paths write a durable, race-free marker:
- Auto-restart: `[agent-procs] Restarted (exit N, attempt M/K)`
- Watch-restart: `[agent-procs] File changed, restarted`
- Manual restart: `[agent-procs] Restarted (manual)`

### Files Changed (Restart Markers)

- `src/daemon/process_manager.rs`: Add `initial_annotation: Option<String>` parameter to `spawn_process()`. Pass it through to `capture_output()`.
- `src/daemon/log_writer.rs`: `capture_output()` accepts `initial_annotation: Option<String>`. When `Some`, writes the line + LIDX entry before the `select!` loop.
- `src/daemon/actor.rs`: All spawn call sites pass the appropriate annotation string. Remove post-spawn supervisor channel sends for restart annotations (replaced by initial_annotation).

### Timestamp Resolution

Log files don't have wall-clock timestamps â€” the LIDX index stores byte offsets and sequence numbers, not times. Resolution strategy depends on the spec type.

**Key invariant:** Both stdout and stderr share the same per-process `AtomicU64` sequence counter. A restart marker written to stdout at sequence number S can be correlated to stderr position by finding the first stderr line with sequence >= S via the LIDX index.

**`restart`:** `--since restart` means "since the last restart annotation." Resolution: scan backward through stdout across all segments (current + rotated) searching for the last line matching the `[agent-procs] Restarted` or `[agent-procs] File changed, restarted` prefix. Record the marker's LIDX sequence number. Serve from the line after that marker forward.

This correctly handles size-based rotation â€” even if the marker has been rotated into `.1`/`.2`, the backward scan finds it.

**Cross-stream resolution:** The marker always lives in stdout (supervisor channel writes to stdout only). When the user requests `--stderr --since restart`:
1. Scan stdout for the marker, record its LIDX sequence number S
2. In stderr's LIDX index, find the first line with sequence >= S
3. Serve stderr from that line forward

The shared sequence counter guarantees that sequence S in stdout corresponds to the same point in wall-clock time as sequence S in stderr.

**Edge cases:**
- **No marker found:** Process has never been restarted. `--since restart` is equivalent to `--since start` (serve everything).
- **Multiple restarts in history:** The backward scan finds the LAST marker, giving output from the most recent restart only.

**`start`:** Serve from line 0 of all segments (current + rotated) â€” the full retained history for this managed process, including output from previous incarnations that survived in rotated segments. Equivalent to no `--since` flag but semantically useful for agents that want to be explicit.

**Duration specs (`30s`, `5m`, `2h`):** Best-effort estimation within the **current incarnation** â€” all segments after the last restart marker (or all segments if no marker exists).

Resolution:
1. Scan stdout backward for the last restart marker. If found, record its position as the incarnation boundary. If not found, the incarnation spans all segments.
2. Count lines from the incarnation boundary to the end across all segments: this is `incarnation_lines`.
3. Get `uptime_secs` from `status --json` (`ProcessInfo.uptime_secs`).
4. If `uptime_secs` is `None` (process exited/failed): fall back to serving everything from the incarnation boundary with a stderr warning `"warning: --since duration is approximate for exited processes"`.
5. If `uptime_secs <= duration_secs`: serve everything from the incarnation boundary.
6. Estimate starting line: `incarnation_boundary + incarnation_lines * (1 - duration_secs / uptime_secs)`.
7. Clamp to valid range.

This correctly handles size-based rotation within a single process lifetime â€” if recent output has been rotated into `.1`, the incarnation line count includes those segments, and the estimate can point into them.

There is no refinement step â€” the LIDX index has no timestamps. For bursty output the estimate may include more lines than requested, which is safe (extra context, not missing data).

**`--since` with `--follow`:** Ignored. The client emits a warning on stderr: `"warning: --since is ignored in follow mode"`. The `since` field is not sent to the daemon.

**Future enhancement (v0.8+):** A `.tidx` timestamp index file could provide exact time-based lookups. The `--since` flag semantics don't change â€” only the resolution accuracy improves.

### Files Changed

- `src/main.rs`: Add `--since` arg.
- `src/cli/logs.rs`: Parse `--since`, resolve to starting line/sequence number client-side. For `restart`, scan stdout backward for marker, use sequence number for cross-stream resolution. For durations, estimate within incarnation boundary.
- `disk_log_reader.rs`: Add backward scan for marker string (across segments). Add incarnation line count (lines after a given position across segments). Add sequence-number-based position lookup for cross-stream resolution.
- `src/daemon/actor.rs`: Emit `[agent-procs] Restarted (manual)` in the `PmCommand::Restart` handler after successful restart.
- No protocol changes needed â€” `--since` is resolved entirely client-side.

## Feature 3: `--context` Flag

### CLI Surface

```bash
# 3 lines before and after each match
agent-procs logs server --grep "panic" --context 3

# In text output, groups separated by --
agent-procs logs server --grep "error" --context 2
# server:140: request received from 10.0.0.1
# server:141: processing order #1234
# server:142: [ERROR] database connection refused
# server:143: retrying in 5s
# server:144: retry attempt 2/5
# --
# server:289: [ERROR] timeout waiting for response
# server:290: circuit breaker tripped
# server:291: returning 503

# In JSON (JSONL format), context lines marked
agent-procs logs server --grep "error" --context 1 --json
# {"process":"server","line_number":141,"stream":"stdout","text":"processing order #1234","context":true}
# {"process":"server","line_number":142,"stream":"stderr","text":"[ERROR] database connection refused"}
# {"process":"server","line_number":143,"stream":"stdout","text":"retrying in 5s","context":true}
```

### Semantics

- `--context N` requires `--grep`. Using `--context` without `--grep` is a parse-time error: `"error: --context requires --grep"` (exit code 1).
- Context lines that are themselves matches are not duplicated.
- Overlapping context windows are merged into a single group.
- Context lines don't count toward `--tail N` â€” only actual matches count. Context is expanded AFTER windowing (step 4 in the filter pipeline).
- **`--follow` mode:** `--context` is not applied (live streaming doesn't have "before" context). The client emits a warning on stderr: `"warning: --context is ignored in follow mode"`. The `context` field is not sent to the daemon.

### Implementation

Runs client-side as step 4 of the filter pipeline. After collecting the windowed match line numbers, expand each match to a range `[match - N, match + N]`, merge overlapping ranges, then read those line ranges from `DiskLogReader`. Tag each line as match or context.

### Files Changed

- `src/main.rs`: Add `--context` arg.
- `src/cli/logs.rs`: Context expansion logic after match collection.

## Feature 4: `--json` Output for Log Lines

### Current State

`logs` currently outputs raw text lines. `--json` does not exist on the `logs` command (only on `status`).

### JSON Format

When `--json` is passed, each line is a JSON object, one per line (JSONL format â€” not a JSON array):

```jsonl
{"process":"server","line_number":142,"stream":"stderr","text":"[ERROR] connection refused"}
{"process":"server","line_number":143,"stream":"stdout","text":"retrying in 5s","context":true}
```

Fields:

| Field | Type | Present | Description |
|-------|------|---------|-------------|
| `process` | string | always | Process name |
| `line_number` | number | non-follow | Line number in the stream's address space. Omitted in `--follow` mode (not available from broadcast stream). |
| `stream` | string | always | `"stdout"` or `"stderr"` |
| `text` | string | always | The log line content |
| `context` | bool | when true | Present and `true` for context lines. Never present in `--follow` mode. |

JSONL (newline-delimited JSON) is chosen over a JSON array because:
- It works with `--follow` streaming (no need to buffer the entire result)
- Each line is independently parseable
- The agent can process lines as they arrive

**Follow mode JSON:** In `--follow --json` mode, the client formats each `Response::LogLine` it receives from the daemon. `line_number` is omitted (the broadcast stream doesn't carry line position). `context` is never present (context is not applied in follow mode). The existing `Response::LogLine` fields (`process`, `stream`, `line`) provide all other needed data.

**Non-follow mode JSON:** The client formats each line from `DiskLogReader` results. `line_number` is available from the read position. `context` is set based on the match/context tagging from the filter pipeline.

### Files Changed

- `src/main.rs`: Add `--json` to `Logs` command.
- `src/cli/logs.rs`: Format output as JSONL when `--json` is set. For follow mode, format from `Response::LogLine`. For non-follow, format from `DiskLogReader` results.

## Feature 5: TUI Regex Filter

### Current State

The TUI filter (`/`) does substring matching via `FilteredIndex`. The index is built by scanning disk lines with `str::contains()`.

### Change

Add regex mode toggle: `Ctrl-R` while in filter input mode switches between substring and regex. The filter bar indicates the current mode:

- Substring mode: `/ error` (current behavior)
- Regex mode: `/r error|panic|warn`

### Behavior

- Default mode is substring (preserves current behavior).
- `Ctrl-R` toggles the mode indicator. The toggle is only available during filter input (`InputMode::FilterInput`).
- Mode resets to substring when the filter is cleared (`Esc`).
- When regex mode is active, `FilteredIndex` uses `Regex::is_match()` instead of `str::contains()`.
- If the regex doesn't compile, the filter bar shows an error indicator and no filtering is applied (the user can keep editing).

### Implementation

- Add `filter_regex: bool` field to `App`.
- Add `FilterAction::ToggleRegex` variant to the `FilterAction` enum.
- In `handle_filter_key` (`src/tui/input.rs`), match `KeyCode::Char('r')` with `KeyModifiers::CONTROL` **before** the generic `Char(c)` arm to emit `FilterAction::ToggleRegex` instead of inserting a literal 'r'.
- `build_filtered_index()` and `update_filtered_index()` check `filter_regex` to decide matching strategy.
- `scan_matching_lines` / `scan_matching_lines_from` in `DiskLogReader` accept a `MatchMode` enum (`Substring(String)` / `Regex(regex::Regex)`) instead of a plain `&str` filter.

### Files Changed

- `src/tui/app.rs`: Add `filter_regex` field, toggle logic. Update `build_filtered_index()` and `update_filtered_index()` to construct `MatchMode::Substring` or `MatchMode::Regex` from filter text + `filter_regex` flag.
- `src/tui/input.rs`: Add `FilterAction::ToggleRegex`. Handle `Ctrl-R` with modifier check before `Char(c)`.
- `src/tui/ui.rs`: Show `/r` indicator for regex mode.
- `disk_log_reader.rs` (shared location after Feature 1 move): Change `scan_matching_lines`/`scan_matching_lines_from` to accept `MatchMode` enum instead of `&str`.

## Testing Strategy

### Unit Tests

**Filter pipeline (`src/cli/logs.rs` or new `src/cli/log_search.rs`):**
- Substring grep returns only matching lines
- Regex grep returns only matching lines
- Invalid regex with `--regex` flag returns error
- `--tail N` + `--grep` returns last N matches (not last N lines filtered)
- `--context N` expands matches with surrounding lines
- Context windows merge when overlapping
- Context lines don't count toward `--tail N`
- Empty results when no matches (not an error)

**Time resolution:**
- `--since restart` scans stdout backward across segments and finds the restart marker
- `--since restart` with no marker returns everything (no restart occurred)
- `--since restart` with `--stderr` resolves via stdout marker sequence number, maps to stderr position
- `--since start` resolves to line 0 of all segments
- `--since 30s` estimates correct line range within incarnation boundary (lines after last marker)
- `--since 30s` when `uptime_secs <= duration_secs` returns everything from incarnation boundary
- `--since 30s` when `uptime_secs` is `None` (exited process) serves everything with warning
- Duration estimation spans rotated segments when incarnation output has been size-rotated
- `--context 3` without `--grep` returns parse-time error

**Restart markers:**
- Manual restart emits `[agent-procs] Restarted (manual)` annotation

**Protocol:**
- `Request::Logs` with new fields round-trips correctly
- Backward compat: old `Logs` request without new fields parses with defaults

**TUI (`app.rs`):**
- `filter_regex` toggle changes matching behavior
- Regex filter builds correct `FilteredIndex`
- Invalid regex in TUI doesn't crash (no filtering applied)

**DiskLogReader:**
- `scan_matching_lines` with `MatchMode::Substring` matches correctly
- `scan_matching_lines` with `MatchMode::Regex` matches correctly

## Explicitly Out of Scope (v0.7)

- Timestamp index file (`.tidx`) â€” future enhancement for precise time-based queries
- Log level awareness / structured log parsing (ERROR/WARN/INFO detection)
- `search` as a separate subcommand
- Cross-session log search
- Cross-process chronological interleaving (requires shared sequence counter)
- Persistent saved searches or bookmarks
- Case-insensitive matching flag (`--ignore-case`) â€” can be added later, not needed for v0.7
- `--before` / `--after` as separate context flags (grep -B / -A) â€” `--context` (symmetric) is sufficient
- Single-process "both streams" mode on CLI (TUI-only via `e` key)
