# agent-procs v0.6 Design Spec

Restart policies, file-watch mode, and filtered disk scrollback.

## Goals

1. Add restart policies so processes can auto-recover from crashes without agent intervention
2. Add file-watch mode so processes restart automatically when source files change
3. Fix filtered disk scrollback so TUI filtering works over the full output history, not just the 10k-line hot buffer

## Theme

Evolve agent-procs from "process launcher" to "process supervisor." The daemon takes responsibility for maintaining desired state — agents can fire-and-forget instead of babysitting.

## Feature 1: Restart Policies

### Data Model

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct RestartPolicy {
    pub mode: RestartMode,
    pub max_restarts: Option<u32>,  // None = unlimited
    pub restart_delay_ms: u64,      // default 1000
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum RestartMode {
    Always,    // restart on any exit
    OnFailure, // restart only on non-zero exit
    Never,     // don't restart (default, current behavior)
}
```

New process state:

```rust
#[derive(Debug, Clone, PartialEq, Serialize)]
#[serde(rename_all = "lowercase")]
pub enum ProcessState {
    Running,
    Exited,
    Failed,   // NEW: max_restarts exhausted
    Unknown,  // NEW: forward-compatibility catch-all
}
```

**Forward compatibility via custom Deserialize:** `#[serde(other)]` cannot be used here because `ProcessState` is an externally tagged string enum (not internally tagged like `Request`/`Response`). Instead, derive `Serialize` for the standard `rename_all` behavior and hand-write `Deserialize` to map unknown strings to `Unknown`:

```rust
impl<'de> Deserialize<'de> for ProcessState {
    fn deserialize<D>(deserializer: D) -> Result<Self, D::Error>
    where D: serde::Deserializer<'de> {
        let s = String::deserialize(deserializer)?;
        Ok(match s.as_str() {
            "running" => Self::Running,
            "exited" => Self::Exited,
            "failed" => Self::Failed,
            _ => Self::Unknown,
        })
    }
}
```

The custom `Deserialize` handles two scenarios:
- **v0.6 clients** correctly map `"failed"` to `Failed` and any future unknown state strings to `Unknown`.
- **Pre-v0.6 clients** use the current derived `Deserialize` which has no `Failed` or `Unknown` variant — they will fail to deserialize `"failed"`. In practice this is unlikely since the daemon auto-spawns from the same binary (see version skew note below).

Extended `ManagedProcess`:

```rust
pub struct ManagedProcess {
    // ...existing fields...
    pub restart_policy: Option<RestartPolicy>,
    pub watch_config: Option<WatchConfig>,
    pub restart_count: u32,
    pub manually_stopped: bool,
    pub restart_pending: bool,
    pub failed: bool,  // NEW: true when max_restarts exhausted or respawn fails
}
```

**Why `failed` is needed:** The current `build_process_infos()` derives `ProcessState` solely from `child.is_some()` — it maps to `Running` or `Exited`. Without an explicit flag, tombstone records and exhausted-restart processes are indistinguishable from normal exits. With `failed`, the mapping becomes:

- `child.is_some()` → `Running`
- `!child.is_some() && failed` → `Failed`
- `!child.is_some() && !failed` → `Exited`

`failed` is set when: max_restarts is exhausted, or `respawn_in_place()` spawn fails. Cleared on: manual `restart`, fresh `run`, or watch-triggered restart.

Key invariant: `manually_stopped` is set by `stop`, cleared by `restart` or a fresh `run`. This is how `stop` disables autorestart without needing a `--no-restart` flag.

```rust
    pub restart_count: Option<u32>,
    pub max_restarts: Option<u32>,       // NEW: needed for "2/5" display

```

`max_restarts` is included so clients can render `restart_count/max_restarts` (e.g., "2/5") deterministically without needing the full `RestartPolicy` struct.

`ProcessState::Failed` and `ProcessState::Unknown` are added to the enum. `Unknown` uses a custom `Deserialize` impl (not `#[serde(other)]`, which doesn't work on externally tagged enums). Protocol version stays at 1 — all new `Request`/`Response` fields use `serde(default)`.

**Version skew note:** The custom `Deserialize` on `ProcessState` only exists in v0.6+ binaries. Pre-v0.6 CLI/TUI binaries will fail to deserialize a `"failed"` state from a v0.6 daemon. In practice this is unlikely: the daemon is auto-spawned from the same binary on first CLI invocation, so daemon and CLI are almost always the same version. If version skew matters in the future, the daemon could map `Failed` to `Exited` when responding to a pre-v0.6 `Hello` handshake.

### Restart Logic in the Daemon

The restart logic extends the actor's existing `refresh_exit_states()` tick (runs every 200ms).

When a newly-exited process is detected:

1. `manually_stopped == true` -> skip
2. `restart_policy` is `None` or `Never` -> skip
3. `restart_policy` is `OnFailure` and `exit_code == Some(0)` -> skip (clean exit)
4. `max_restarts` reached -> set `failed = true`, emit `[agent-procs] Max restarts (5) exhausted` through supervisor channel (capture task is still alive at this point since the process just exited), skip
5. Otherwise -> eligible for restart

Eligible processes get a delayed restart: set `restart_pending = true` on the process record, then `tokio::spawn` a task that sleeps for `restart_delay_ms` and sends `PmCommand::AutoRestart { name }` through the actor channel. The `restart_pending` flag prevents the next 200ms tick from scheduling a duplicate restart for the same crash. The flag is cleared when `AutoRestart` fires (whether it succeeds or fails), or when the process is manually stopped/restarted/removed.

`collect_restartable_processes()` skips processes where `restart_pending == true`.

`PmCommand::AutoRestart` is internal-only (not exposed in wire protocol). It:

1. Re-checks that the process still exists in the map, is still dead, `manually_stopped` is still false, and `restart_pending` is still true (guard against race with manual `restart`, `stop`, or `stop-all` during the delay)
2. Increments `restart_count`
3. Calls `respawn_in_place()` (see below) — which drains the capture task, rotates logs, and attempts re-spawn
4. **If spawn succeeded:** sends `[agent-procs] Restarted (exit 1, attempt 2/5)` through the NEW process's supervisor channel — it becomes the first line in the fresh log segment, written to disk by the new capture task
5. **If spawn failed:** broadcasts `[agent-procs] Restart failed: <error>` through `output_tx` only (visible in `--follow` and live TUI). No disk write — there is no capture task. The `Failed` state in `status` output is the durable record.
6. Clears `restart_pending`

**Why annotations go AFTER spawn, not before:** The annotation must reflect what actually happened. Sending "Restarted" before the spawn attempt risks a false-success log line if spawn fails. And once the old capture tasks are drained (step 2 of `respawn_in_place()`), there is no task to write to disk. The new capture task (created by successful spawn) is the correct destination for the success annotation.

**`respawn_in_place()`**: A new method on `ProcessManager` that handles the re-spawn path for both AutoRestart and WatchRestart. It cannot reuse `spawn_process()` directly because `spawn_process()` rejects duplicate names (line 119 of `process_manager.rs`), and the current `restart_process()` works around this by removing the old record first (line 317-318). `respawn_in_place()`:

1. Extracts and saves supervisor metadata from the existing record (restart_policy, watch_config, restart_count, manually_stopped, failed) and original spawn args (command, cwd, env, port)
2. Drops the supervisor channel sender (signals the capture task to drain and exit)
3. Awaits the stored `JoinHandle`s for the capture tasks (ensures all pending output is flushed to disk and LIDX)
4. Rotates current log files using the existing `rotate_log_files()` for both `.stdout` and `.stderr` — this preserves pre-restart output in rotated segments that `DiskLogReader` already knows how to read across
5. Removes the old process record
6. Calls `spawn_process()` with the original command/cwd/env/port (which creates fresh log files via `File::create`)
7. **If spawn succeeds:** copies saved supervisor metadata onto the new record, clears `failed`. Returns Ok.
8. **If spawn fails:** reinserts a tombstone record with `child: None`, `exit_code: None`, `failed: true`, and the saved supervisor metadata. This ensures the process remains visible in `status` output and the TUI with a `Failed` state, rather than silently disappearing. Returns Err with the error message.

The rotation step (4) is critical: without it, `File::create` in `capture_output()` truncates the log, erasing all pre-restart output and the supervisor annotation. The existing `rotate_log_files()` and `DiskLogReader` segment-aware reading handle this correctly.

This keeps `spawn_process()` unchanged (it remains the clean "create new process" path) while giving the supervisor a proper re-spawn path that preserves state and handles failures gracefully.

**Synthetic log line plumbing (per-process supervisor channel):** Supervisor events (restart, failure) need to appear in both live streams AND on-disk history. Broadcasting through `output_tx` alone is not sufficient — disk writes only happen inside `capture_output()`, which reads from child process pipes. A supervisor line broadcast through `output_tx` would appear in `--follow` and the live TUI, but NOT in `logs --tail`, TUI cold-start history, or filtered disk scrollback.

A standalone `write_supervisor_line()` that reopens the log file is also unsafe — `capture_output()` owns the file handle, `IndexWriter`, and `bytes_written` offset. An out-of-band append would leave that state stale and corrupt subsequent LIDX entries.

**Solution: per-process supervisor channel.** Route supervisor lines through the same `capture_output()` task that owns the file state:

1. When spawning a process, create an `mpsc::Sender<String>` per stream (stdout only for supervisor lines). Store the sender in `ManagedProcess`. Pass the receiver to `capture_output()`.
2. `capture_output()` is extended to `select!` between the pipe reader and the supervisor channel. Both paths write through the same file handle / index writer / bytes_written tracking.
3. After the pipe hits EOF (process exited), `capture_output()` continues draining the supervisor channel until it's closed. This is the key insight: the capture task outlives the process pipe.
4. The actor writes supervisor lines by sending through the stored `mpsc::Sender`.
5. When the process is removed from the map, the sender is dropped, which closes the channel, which terminates the capture task.

Store the `JoinHandle`s from `tokio::spawn(capture_output(...))` in `ManagedProcess` (two handles: stdout + stderr). `respawn_in_place()` needs these to synchronize before rotating logs (see Feature 1: Respawn Log Rotation below).

This ensures all writes are serialized in a single task, maintaining correct byte offsets and LIDX state. Supervisor annotations are visible everywhere: `logs --tail`, `logs --follow`, TUI live view, TUI disk scrollback, and filtered disk scrollback.

`restart_count` resets to 0 on manual `restart`, fresh `run`, or watch-triggered restart.

## Feature 2: File Watch Mode

### Data Model

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct WatchConfig {
    pub paths: Vec<String>,
    pub ignore: Option<Vec<String>>,
}
```

### Watcher Architecture

```
notify::RecommendedWatcher
    -> events -> mpsc channel -> debounce task (500ms)
                                     -> PmCommand::WatchRestart { name }
                                          -> actor channel
```

Each watched process gets:

1. A `notify::RecommendedWatcher` subscribed to the configured paths
2. A debounce task (`tokio::spawn`) that collects file events, waits 500ms of quiet, then sends `PmCommand::WatchRestart` to the actor

### Path Resolution

Watch paths are glob patterns relative to the process's `cwd` (or daemon's cwd if none set). The daemon expands globs at watcher setup time to find directories to watch, then filters incoming events against the glob patterns using `globset`.

### Default Ignore List

Always ignored regardless of user config: `.git`, `node_modules`, `target`, `__pycache__`, `*.pyc`, `.DS_Store`. User-provided `ignore` patterns are additive.

### Watch Restart Behavior

`PmCommand::WatchRestart` in the actor:

1. Checks process still exists and is not `manually_stopped` — if either, ignores the event
2. Sends supervisor line through the process's supervisor channel: `[agent-procs] File changed: src/main.rs, restarting`
3. Stops the process (without setting `manually_stopped`) — sends SIGTERM/SIGKILL as usual
4. Calls `respawn_in_place()` which: drains capture tasks → rotates logs → re-spawns
5. Resets `restart_count` to 0 (watch restarts are intentional, not crashes)
6. Re-creates the file watcher for the new process instance

### Watch + Stop Interaction

`stop` tears down the watcher and sets `manually_stopped`. The process is fully stopped — no autorestart, no watch restarts. `restart` clears `manually_stopped`, re-spawns the process, and re-creates the watcher if the process has a `WatchConfig`.

Rationale: `stop` means stop. No lingering watchers, no suppressed events, no hidden state.

### Watch + Restart Policy Interaction

They are independent:

- Process crashes -> restart policy handles it (with delay and counting)
- File changes -> watch handler restarts immediately (no delay, no counting)
- File changes while process is in `Failed` state (crashed, restarts exhausted) -> watch restart fires, clears failure, resets `restart_count`
- Process was `stop`ped -> watcher is torn down, no file events received

### Watcher Survives Process Exit (Crash)

The file watcher stays active when a process exits on its own (crash or clean exit). A file change on a crashed process restarts it. This handles the workflow: server crashes, agent edits code to fix the bug, watch detects the change and restarts automatically.

### Watcher Lifecycle

- **Created:** when a process is spawned with a `WatchConfig`
- **Torn down on:** explicit `stop`, `stop-all`, or daemon shutdown
- **Survives:** process crash/exit (not manually stopped)
- **Re-created on:** `restart` (if process has a `WatchConfig`)

### Files Changed

- `Cargo.toml`: Add `notify = "7"` and `globset = "0.4"` dependencies.
- `src/daemon/watcher.rs` (new): File watcher setup, debounce task, glob matching, default ignore list.
- `src/daemon/actor.rs`: Add `PmCommand::WatchRestart`. Store watcher handles alongside process state. Create/drop watchers on spawn/remove.
- `src/daemon/mod.rs`: Add `mod watcher`.

## Feature 6: Filtered Disk Scrollback

### Problem

When a filter is active, `visible_lines()` returns `None` and the TUI falls back to collecting all matching lines from the 10k-line hot buffer, then using `Paragraph::scroll()`. Filtering only covers recent output, not the full disk history.

### Solution

When a filter is applied, scan the disk logs and build a cached index of matching line numbers. Use the index for windowed rendering.

### FilteredIndex

```rust
struct FilteredIndex {
    filter: String,
    stream_mode: StreamMode,
    matching_lines: Vec<usize>,
    scanned_up_to: usize,
}
```

Stored per-process in `App` as `HashMap<String, FilteredIndex>`. The index is keyed by both the filter string and the stream mode — changing either triggers a rebuild. This is necessary because "Both" mode interleaves stdout/stderr by sequence number, producing different line numbering than stdout-only or stderr-only.

### Lifecycle

1. User sets filter -> scan disk logs line-by-line, collect indices of lines containing the filter substring -> store as `FilteredIndex`
2. `visible_lines()` with filter uses `matching_lines` as a virtual address space: "line 500 of filtered results" maps to `matching_lines[500]`, which gives the actual line number for a disk seek via the LIDX index
3. New output arrives -> incrementally scan only new lines, append matches to the index
4. Filter changes -> rebuild from scratch
5. Stream mode changes (`e` key) -> rebuild from scratch (different line numbering)
6. Filter cleared -> drop the index, return to normal windowed rendering

### visible_lines() Change

Instead of returning `None` when filtered, `visible_lines()` uses the `FilteredIndex` to compute the window, then reads specific lines from `DiskLogReader` by line number. The `DiskLogReader` already supports line-by-line access via LIDX.

### DiskLogReader Additions

The scan and read APIs must be stream-mode-aware, because "Both" mode interleaves stdout/stderr by sequence number, producing different line numbering than single-stream modes.

**Scanning:**
- `scan_matching_lines(filter: &str, mode: StreamMode) -> Vec<usize>` — scans through lines in the given mode's ordering and returns indices of matches. For `Stdout`/`Stderr`, scans the single stream. For `Both`, scans the interleaved sequence and returns indices into the interleaved address space.

**Reading:**
- `read_scattered_lines(mode: StreamMode, line_numbers: &[usize]) -> Vec<(LineSource, String)>` — reads non-contiguous lines by index within the given mode's address space. Each line number is resolved to a byte offset via the LIDX index, then read individually. The segment cache (500ms TTL) prevents repeated stat calls.

This ensures the `FilteredIndex` (which is keyed by stream mode) can correctly build and read indices regardless of which mode is active.

### Performance

A 50MB log file is roughly 500k-1M lines. Sequential scan for substring matching is fast (tens of milliseconds). The scan runs synchronously on filter confirm. Incremental updates scan only new lines as they arrive.

### Hot Buffer Interaction

When the filter is active, new lines still flow into the hot buffer. Incremental scan checks new lines against the filter and appends to `matching_lines`. Windowed rendering reads recent matches from the hot buffer using the same disk/hot boundary logic as unfiltered mode.

## Testing Strategy

### Unit Tests

- `RestartPolicy` / `WatchConfig` / `RestartMode` serde roundtrips
- `ProcessState::Failed` and `Unknown` serde roundtrips; verify unknown state strings deserialize to `Unknown` via custom Deserialize impl (not `#[serde(other)]`)
- `collect_restartable_processes()` logic: each decision branch (manually stopped, restart_pending, clean exit with on-failure, max restarts exhausted, eligible)
- `respawn_in_place()` preserves supervisor metadata across re-spawn
- Config parsing with new fields: verify existing configs still parse, new fields deserialize correctly
- `capture_output()` supervisor channel: verify lines sent through supervisor channel appear in log file, LIDX index, and broadcast
- `respawn_in_place()` rotates logs before re-spawn: verify pre-restart output survives in rotated segments
- `FilteredIndex` build and incremental update, per-stream-mode correctness
- `scan_matching_lines()` and `read_scattered_lines()` with all three stream modes

### Integration Tests

- **Restart on crash**: process exits non-zero with `autorestart: on-failure`, verify restart and `restart_count` increment
- **Max restarts exhausted**: `max_restarts: 2`, verify state becomes `Failed` after 2 restarts
- **On-failure skips clean exit**: process exits 0 with `autorestart: on-failure`, verify no restart
- **Stop disables autorestart**: start with `autorestart: always`, stop, verify stays dead
- **Restart clears manually_stopped**: stop, then `restart`, verify autorestart is active again
- **Watch triggers restart**: long-running process with `--watch`, touch watched file, verify restart and synthetic log line
- **Watch ignores ignored patterns**: touch ignored file, verify no restart
- **Watch restarts dead process**: let process die, touch watched file, verify restart
- **Restart delay honored**: `restart_delay: 2000`, verify ~2s gap between exit and restart
- **No duplicate restarts**: `restart_delay: 2000` with `max_restarts: 1`, process crashes, verify exactly 1 restart (not multiple from repeated tick detection)
- **Supervisor lines on disk**: restart a process, then read logs via `logs --tail`, verify `[agent-procs] Restarted` line appears in disk output
- **Pre-restart output preserved**: start process with output, trigger autorestart, verify pre-restart output is readable via `logs --tail` (in rotated segment)

### Not Tested Automatically

- `notify` crate internals (well-tested upstream)
- Exact debounce timing (flaky)
- TUI rendering of new indicators (manual, consistent with existing approach)
- Filtered disk scrollback rendering (manual; unit tests cover the index logic)

## Explicitly Out of Scope (v0.6)

- Exponential backoff restarts (fixed delay is sufficient for now)
- Health check probes (HTTP/TCP liveness checks)
- MCP server integration
- stdin forwarding / interactive process access
- Procfile format support
- Process scaling / formation (`numprocs`)
- Configurable stop signal
- Regex filtering in TUI (substring matching is sufficient)
- Remote/programmatic API
