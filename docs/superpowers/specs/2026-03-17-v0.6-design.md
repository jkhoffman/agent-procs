# agent-procs v0.6 Design Spec

Restart policies, file-watch mode, and filtered disk scrollback.

## Goals

1. Add restart policies so processes can auto-recover from crashes without agent intervention
2. Add file-watch mode so processes restart automatically when source files change
3. Fix filtered disk scrollback so TUI filtering works over the full output history, not just the 10k-line hot buffer

## Theme

Evolve agent-procs from "process launcher" to "process supervisor." The daemon takes responsibility for maintaining desired state — agents can fire-and-forget instead of babysitting.

## Feature 1: Restart Policies

### Data Model

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct RestartPolicy {
    pub mode: RestartMode,
    pub max_restarts: Option<u32>,  // None = unlimited
    pub restart_delay_ms: u64,      // default 1000
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum RestartMode {
    Always,    // restart on any exit
    OnFailure, // restart only on non-zero exit
    Never,     // don't restart (default, current behavior)
}
```

New process state:

```rust
pub enum ProcessState {
    Running,
    Exited,
    Failed,  // NEW: max_restarts exhausted
}
```

Extended `ManagedProcess`:

```rust
pub struct ManagedProcess {
    // ...existing fields...
    pub restart_policy: Option<RestartPolicy>,
    pub watch_config: Option<WatchConfig>,
    pub restart_count: u32,
    pub manually_stopped: bool,
}
```

Key invariant: `manually_stopped` is set by `stop`, cleared by `restart` or a fresh `run`. This is how `stop` disables autorestart without needing a `--no-restart` flag.

### Protocol Changes

`Request::Run` gets two new optional fields:

```rust
Request::Run {
    command: String,
    name: Option<String>,
    cwd: Option<String>,
    env: Option<HashMap<String, String>>,
    port: Option<u16>,
    restart: Option<RestartPolicy>,   // NEW
    watch: Option<WatchConfig>,       // NEW
}
```

Both use `#[serde(default)]` for forward compatibility.

`ProcessInfo` gets new display fields:

```rust
pub struct ProcessInfo {
    // ...existing fields...
    pub restart_count: Option<u32>,
    pub restart_policy: Option<String>,  // "always", "on-failure", or absent
    pub watched: Option<bool>,
}
```

`ProcessState::Failed` is added to the enum. Protocol version stays at 1 — all changes are additive with `serde(default)`.

### Restart Logic in the Daemon

The restart logic extends the actor's existing `refresh_exit_states()` tick (runs every 200ms).

When a newly-exited process is detected:

1. `manually_stopped == true` -> skip
2. `restart_policy` is `None` or `Never` -> skip
3. `restart_policy` is `OnFailure` and `exit_code == Some(0)` -> skip (clean exit)
4. `max_restarts` reached -> set state to `Failed`, emit synthetic log line, skip
5. Otherwise -> eligible for restart

Eligible processes get a delayed restart: `tokio::spawn` a task that sleeps for `restart_delay_ms`, then sends `PmCommand::AutoRestart { name }` through the actor channel.

`PmCommand::AutoRestart` is internal-only (not exposed in wire protocol). It:

1. Re-checks that the process still exists in the map, is still dead, and `manually_stopped` is still false (guard against race with manual `restart`, `stop`, or `stop-all` during the delay)
2. Increments `restart_count`
3. Calls `spawn_process()` with the same command/cwd/env/port
4. Emits synthetic log line: `[agent-procs] Restarted (exit 1, attempt 2/5)`
5. If spawn fails, emits `[agent-procs] Restart failed: <error>` and sets state to `Failed`

Synthetic log lines go through the existing `output_tx` broadcast channel as `OutputLine` with `stream: Stdout`. They appear in `logs --follow`, the TUI, and on-disk log files with no special plumbing.

`restart_count` resets to 0 on manual `restart`, fresh `run`, or watch-triggered restart.

- `src/protocol.rs`: Add `RestartPolicy`, `RestartMode`, `WatchConfig` structs. Add `Failed` to `ProcessState`. Add `restart_count`, `restart_policy`, `watched` to `ProcessInfo`. Add `restart`, `watch` to `Request::Run`.

## Feature 2: File Watch Mode

### Data Model

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct WatchConfig {
    pub paths: Vec<String>,
    pub ignore: Option<Vec<String>>,
}
```

### Watcher Architecture

```
notify::RecommendedWatcher
    -> events -> mpsc channel -> debounce task (500ms)
                                     -> PmCommand::WatchRestart { name }
                                          -> actor channel
```

Each watched process gets:

1. A `notify::RecommendedWatcher` subscribed to the configured paths
2. A debounce task (`tokio::spawn`) that collects file events, waits 500ms of quiet, then sends `PmCommand::WatchRestart` to the actor

### Path Resolution

Watch paths are glob patterns relative to the process's `cwd` (or daemon's cwd if none set). The daemon expands globs at watcher setup time to find directories to watch, then filters incoming events against the glob patterns using `globset`.

### Default Ignore List

Always ignored regardless of user config: `.git`, `node_modules`, `target`, `__pycache__`, `*.pyc`, `.DS_Store`. User-provided `ignore` patterns are additive.

### Watch Restart Behavior

`PmCommand::WatchRestart` in the actor:

1. Checks process still exists and is not `manually_stopped` — if either, ignores the event
2. Emits synthetic log line: `[agent-procs] File changed: src/main.rs, restarting`
3. Stops the process (without setting `manually_stopped`)
4. Re-spawns with same config
5. Resets `restart_count` to 0 (watch restarts are intentional, not crashes)
6. Re-creates the file watcher for the new process instance

### Watch + Stop Interaction

`stop` tears down the watcher and sets `manually_stopped`. The process is fully stopped — no autorestart, no watch restarts. `restart` clears `manually_stopped`, re-spawns the process, and re-creates the watcher if the process has a `WatchConfig`.

Rationale: `stop` means stop. No lingering watchers, no suppressed events, no hidden state.

### Watch + Restart Policy Interaction

They are independent:

- Process crashes -> restart policy handles it (with delay and counting)
- File changes -> watch handler restarts immediately (no delay, no counting)
- File changes while process is in `Failed` state (crashed, restarts exhausted) -> watch restart fires, clears failure, resets `restart_count`
- Process was `stop`ped -> watcher is torn down, no file events received

### Watcher Survives Process Exit (Crash)

The file watcher stays active when a process exits on its own (crash or clean exit). A file change on a crashed process restarts it. This handles the workflow: server crashes, agent edits code to fix the bug, watch detects the change and restarts automatically.

### Watcher Lifecycle

- **Created:** when a process is spawned with a `WatchConfig`
- **Torn down on:** explicit `stop`, `stop-all`, or daemon shutdown
- **Survives:** process crash/exit (not manually stopped)
- **Re-created on:** `restart` (if process has a `WatchConfig`)

### Files Changed

- `Cargo.toml`: Add `notify = "7"` and `globset = "0.4"` dependencies.
- `src/daemon/watcher.rs` (new): File watcher setup, debounce task, glob matching, default ignore list.
- `src/daemon/actor.rs`: Add `PmCommand::WatchRestart`. Store watcher handles alongside process state. Create/drop watchers on spawn/remove.
- `src/daemon/mod.rs`: Add `mod watcher`.

## Feature 6: Filtered Disk Scrollback

### Problem

When a filter is active, `visible_lines()` returns `None` and the TUI falls back to collecting all matching lines from the 10k-line hot buffer, then using `Paragraph::scroll()`. Filtering only covers recent output, not the full disk history.

### Solution

When a filter is applied, scan the disk logs and build a cached index of matching line numbers. Use the index for windowed rendering.

### FilteredIndex

```rust
struct FilteredIndex {
    filter: String,
    stream_mode: StreamMode,
    matching_lines: Vec<usize>,
    scanned_up_to: usize,
}
```

Stored per-process in `App` as `HashMap<String, FilteredIndex>`. The index is keyed by both the filter string and the stream mode — changing either triggers a rebuild. This is necessary because "Both" mode interleaves stdout/stderr by sequence number, producing different line numbering than stdout-only or stderr-only.

### Lifecycle

1. User sets filter -> scan disk logs line-by-line, collect indices of lines containing the filter substring -> store as `FilteredIndex`
2. `visible_lines()` with filter uses `matching_lines` as a virtual address space: "line 500 of filtered results" maps to `matching_lines[500]`, which gives the actual line number for a disk seek via the LIDX index
3. New output arrives -> incrementally scan only new lines, append matches to the index
4. Filter changes -> rebuild from scratch
5. Stream mode changes (`e` key) -> rebuild from scratch (different line numbering)
6. Filter cleared -> drop the index, return to normal windowed rendering

### visible_lines() Change

Instead of returning `None` when filtered, `visible_lines()` uses the `FilteredIndex` to compute the window, then reads specific lines from `DiskLogReader` by line number. The `DiskLogReader` already supports line-by-line access via LIDX.

### DiskLogReader Addition

Add `read_scattered_lines(source, line_numbers: &[usize]) -> Vec<(LineSource, String)>` for reading non-contiguous lines efficiently. Each line number is resolved to a byte offset via the LIDX index, then read individually. The segment cache (500ms TTL) prevents repeated stat calls.

### Performance

A 50MB log file is roughly 500k-1M lines. Sequential scan for substring matching is fast (tens of milliseconds). The scan runs synchronously on filter confirm. Incremental updates scan only new lines as they arrive.

### Hot Buffer Interaction

When the filter is active, new lines still flow into the hot buffer. Incremental scan checks new lines against the filter and appends to `matching_lines`. Windowed rendering reads recent matches from the hot buffer using the same disk/hot boundary logic as unfiltered mode.

### Files Changed

- `src/tui/app.rs`: Add `FilteredIndex` struct. Add `filtered_indices: HashMap<String, FilteredIndex>` to `App`. Update `visible_lines()` to use filtered index instead of returning `None`. Update `line_count_for()` to use `matching_lines.len()`. Update `confirm_filter()` to trigger index build. Update `push_output()` to incrementally update index.
- `src/tui/disk_log_reader.rs`: Add `read_scattered_lines()` method. Add `scan_matching_lines(filter: &str) -> Vec<usize>` method that scans through all lines and returns indices of matches.
- `src/tui/ui.rs`: Remove the filtered fallback branch in `draw_output()` — `visible_lines()` now handles both cases.

## Testing Strategy

### Unit Tests

- `RestartPolicy` / `WatchConfig` / `RestartMode` serde roundtrips
- `collect_restartable_processes()` logic: each decision branch (manually stopped, clean exit with on-failure, max restarts exhausted, eligible)
- Config parsing with new fields: verify existing configs still parse, new fields deserialize correctly
- `ProcessState::Failed` serde roundtrip
- `FilteredIndex` build and incremental update
- `read_scattered_lines()` correctness

### Integration Tests

- **Restart on crash**: process exits non-zero with `autorestart: on-failure`, verify restart and `restart_count` increment
- **Max restarts exhausted**: `max_restarts: 2`, verify state becomes `Failed` after 2 restarts
- **On-failure skips clean exit**: process exits 0 with `autorestart: on-failure`, verify no restart
- **Stop disables autorestart**: start with `autorestart: always`, stop, verify stays dead
- **Restart clears manually_stopped**: stop, then `restart`, verify autorestart is active again
- **Watch triggers restart**: long-running process with `--watch`, touch watched file, verify restart and synthetic log line
- **Watch ignores ignored patterns**: touch ignored file, verify no restart
- **Watch restarts dead process**: let process die, touch watched file, verify restart
- **Restart delay honored**: `restart_delay: 2000`, verify ~2s gap between exit and restart

### Not Tested Automatically

- `notify` crate internals (well-tested upstream)
- Exact debounce timing (flaky)
- TUI rendering of new indicators (manual, consistent with existing approach)
- Filtered disk scrollback rendering (manual; unit tests cover the index logic)

## Explicitly Out of Scope (v0.6)

- Exponential backoff restarts (fixed delay is sufficient for now)
- Health check probes (HTTP/TCP liveness checks)
- MCP server integration
- stdin forwarding / interactive process access
- Procfile format support
- Process scaling / formation (`numprocs`)
- Configurable stop signal
- Regex filtering in TUI (substring matching is sufficient)
- Remote/programmatic API
